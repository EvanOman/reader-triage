## 1. Chain-of-Thought Before Scoring

### The Evidence

Chain-of-thought (CoT) prompting -- asking the LLM to articulate its reasoning before producing a final score -- is one of the most consistently supported techniques in LLM evaluation research.

**G-Eval** (Liu et al., 2023) demonstrated this with a two-stage approach: first generating evaluation steps via CoT, then using those steps during scoring. The result was a Spearman correlation of 0.514 with human ratings on summarization tasks, outperforming all prior automated methods by a large margin. Ablation analysis confirmed that including CoT steps added 2-3 points to the Spearman correlation coefficient.

**Prometheus** (Kim et al., 2023) trained a 13B-parameter evaluator LLM to generate detailed feedback *before* outputting a score, separated by a `[RESULT]` token. This "feedback-first" approach achieved a Pearson correlation of 0.897 with human evaluators -- on par with GPT-4's 0.882 and far above ChatGPT's 0.392.

**Cameron Wolfe's analysis** (2024) emphasizes a critical ordering requirement: the rationale must come *before* the score, not after. Research shows that "conclusions generated by the model are not supported by the explanation generated afterward" -- when the score comes first, the model tends to rationalize rather than reason.

**Confident AI's research** found that CoT prompting improved consistency from 65.0% to 77.5% in LLM judge evaluations.

### The Mechanism

CoT works for evaluation because it:
- Forces the model to attend to specific evidence in the content before committing to a judgment
- Creates an audit trail for debugging scoring decisions
- Reduces the probability of snap judgments based on surface features (length, tone, vocabulary)
- Helps the model consider conflicting signals rather than anchoring on the first salient feature

### Practical Implications

**Reasoning should precede the final answer in the output structure.** If you ask for JSON with scores and reasons, the model should be prompted to think through the reasoning first, then produce the structured output.

Two patterns work well:

**Pattern A: Embedded reasoning (current approach, modified)**
```
For each dimension, first explain your reasoning, then give your categorical answer.

Respond with JSON where each dimension has a "reason" field before the "answer" field:
{
  "quotability_reason": "The article contains 3 specific data points...",
  "standalone_passages": "several",
  ...
}
```

**Pattern B: Two-phase output (G-Eval style)**
```
First, write a brief evaluation of the article across the four scoring dimensions.
Then, provide your categorical answers as a JSON object.

=== EVALUATION ===
[free-form reasoning here]

=== SCORES ===
{"standalone_passages": "several", ...}
```

Pattern B is more robust because the free-form reasoning section allows the model to explore nuance without being constrained by the JSON structure, but Pattern A is easier to parse and our current approach already approximates it.

## 2. Structured Output Formats

### JSON vs. XML vs. YAML

**JSON is the dominant format** for LLM evaluation outputs, and for good reason:

1. **Native support**: Anthropic, OpenAI, and Google all offer constrained JSON output modes. Claude's structured outputs feature (beta, available for Sonnet 4.5 and Opus 4.1) uses constrained decoding to guarantee valid JSON at the token level -- the model literally cannot produce tokens that violate the schema.

2. **Parsing reliability**: JSON has unambiguous delimiters and well-defined escaping rules. XML requires dealing with entity encoding and closing tags; YAML's whitespace sensitivity makes it fragile with LLM output.

3. **Token efficiency**: JSON is moderately token-efficient. YAML saves some tokens by eliminating braces and quotes, but the whitespace sensitivity creates parsing failures. XML is the most verbose.

4. **LLM familiarity**: LLMs have seen far more JSON in training data than structured XML or YAML evaluation formats. The StructEval benchmark (2025) showed that even state-of-the-art models like o1-mini achieve only 75.58% average score on structured output generation across formats, with JSON being among the most reliable.

**TOON format** (a tabular format with explicit schema declarations) achieved 73.9% accuracy vs. 69.7% for JSON on data retrieval tasks, suggesting that for very structured data, table-like formats may help. However, this is not applicable to our evaluation use case.

### Constrained Decoding vs. Prompt-Only

There are two approaches to enforcing structured output:

1. **Prompt-only**: Include the expected JSON schema in the prompt and hope the model complies. This is what we currently do.
2. **Constrained decoding**: Use the API's structured output mode to guarantee schema compliance at the token level.

Constrained decoding eliminates JSON parsing failures entirely, but has tradeoffs:
- Currently only available in beta for select Claude models
- May slightly reduce reasoning quality if the schema constrains the model's ability to express uncertainty
- Adds complexity to the API call

**For our use case**, prompt-only JSON is sufficient. Our schema is simple (flat object with string/boolean fields), and we already handle markdown code block stripping. The failure rate is low enough that constrained decoding is not worth the added complexity. If parsing failures increase, constrained decoding is a good fallback.

## 3. Few-Shot Examples

### How Many Examples?

The evidence on few-shot examples for LLM evaluation is nuanced and model-dependent:

**For strong models (GPT-4, Claude Sonnet/Opus):** Few-shot examples provide modest improvement. One study found that using few-shot prompts with GPT-4 "didn't make an obvious difference in consistency" and actually "caused slight variance in score ranges." The model is already capable of understanding the rubric from the instructions alone.

**For weaker models (GPT-3.5, Haiku-class):** Few-shot examples are transformative. "Including few examples for GPT-3.5-turbo significantly improves consistency and makes results usable."

**General finding:** "Adding a few examples in the prompts is probably the single most helpful thing one could do to steer LLM-generated outputs" (Confident AI). Few-shot prompting increased GPT-4's consistency from 65.0% to 77.5%.

The Patronus AI guide recommends "5-10 expert examples in your judge's prompt" during initial calibration, then selecting the most representative subset for production.

### How to Select Examples

1. **Cover the score distribution**: Include examples of high, medium, and low-scoring content. If all examples are high-scoring, the model's baseline shifts upward.

2. **Include edge cases**: Pick examples where the correct answer is non-obvious -- content that scores high on one dimension but low on another.

3. **Match the domain**: Use examples from the same content type being evaluated (articles for article scoring, podcasts for podcast scoring).

4. **Balance positive and negative signals**: Avoid skewing toward one end of the scale.

**Warning about instability:** Research from Zhao et al. (2021) found that few-shot performance is highly sensitive to example order, label distribution, and prompt format. "The choice of prompt format, training examples, and even the order of the training examples can cause accuracy to vary from near chance to near state-of-the-art." This was specifically documented for factual consistency tasks where "performance was unstable when changing the label, example order, and number of examples."

### Practical Recommendation

For our Sonnet-based scorer, 2-3 well-chosen examples would be the sweet spot:
- One high-scoring article (60+) with clear quotable passages and frameworks
- One medium-scoring article (30-59) that has some good qualities but lacks depth
- One low-scoring article (<30) that is primarily a news roundup or thin summary

These should be real articles from our dataset where we have high confidence in the correct score.

## 4. Reference-Based Evaluation

### Inline Rubric vs. Few-Shot Examples

There are two complementary approaches to calibrating an LLM judge:

**Inline rubric**: Provide explicit scoring criteria and definitions in the prompt. Define exactly what each score level means ("a_few" = "2-4 passages that could be highlighted," "several" = "5+ passages," etc.).

**Reference examples**: Show the model what correct scoring looks like on known content.

**Research conclusion: use both, but the rubric is primary.** The Prometheus model architecture requires four components in every evaluation prompt: (1) the instruction, (2) the response to evaluate, (3) a reference answer showing ideal scoring, and (4) a custom scoring rubric. The rubric is the foundation; examples calibrate the model's interpretation of the rubric.

The "Rubric Is All You Need" paper (ACM ICER 2025) found that question-specific rubrics improved LLM-based code evaluation significantly, reinforcing that clear rubric definitions are more important than example volume.

### What Makes a Good Rubric?

Research consistently recommends:
- **Specific, unambiguous criteria**: Not "Is this article good?" but "How many passages could stand alone as a saved note?"
- **Defined score levels**: Each possible response option should have a clear definition
- **One dimension per criterion**: Don't combine multiple qualities into a single question

Our current rubric does well on the first point (questions are specific) but could improve on the second (we don't define what "a_few" vs. "several" means numerically) and third (some questions like "content type" are classifications, not evaluations).

## 5. System Prompt vs. User Prompt

### Where to Place the Rubric

The standard best practice uses a two-role structure:

- **System prompt**: Evaluation framework, judge persona, scoring instructions, response format requirements
- **User prompt**: The specific content to evaluate, plus any per-item context (title, author, word count)

**Rationale:** The system prompt establishes a consistent "judge mode" across all evaluations. The rubric and scoring criteria belong here because they are invariant across items. The user prompt contains variable content that changes per evaluation.

**Our current approach** puts everything in the user prompt (a single `messages` call with `role: "user"`). This works but misses an opportunity to leverage the system prompt's privileged position in the model's attention hierarchy. System prompts are processed with higher priority and more consistent attention, making them ideal for instructions that must be followed precisely.

### Recommended Structure

```python
messages = [
    {
        "role": "user",
        "content": """Article Title: {title}
Author: {author}
Word Count: {word_count}

Content:
{content}"""
    }
]
system = """You are an expert content evaluator assessing articles for capture value --
how likely a reader is to want to save and highlight passages.

[Full rubric and scoring criteria here]

[Output format specification here]"""
```

This separation keeps the system prompt stable across API calls (better for caching) and makes the variable content cleanly isolated.

## 6. Temperature Settings

### What Temperature for Evaluation?

The consensus across the literature is clear: **use low temperature for evaluation tasks.**

- **Kinde's guide**: "Use a temperature of 0 or very low values to minimize randomness and ensure deterministic outputs"
- **Patronus AI**: Recommends "lower temps (0.2-0.3) to reduce randomness in evaluations"
- **Evidently AI**: "Set a low temperature" -- for evaluations, you don't need creativity
- **General research**: A temperature between 0.2 and 0.7 "typically yields reliable, relevant output" for instruction-following tasks

### The Temperature 0 Caveat

Even at temperature 0, LLMs are not truly deterministic. Multiple factors contribute to non-determinism:
- GPU floating-point arithmetic varies between runs
- Batching and parallelism in inference affect computation order
- API providers may route to different hardware

One OpenAI engineer noted that "their models do not output identical token scores between runs for the same input." The `seed` parameter can improve reproducibility but does not guarantee it.

### Temperature and Score Distribution

A subtle finding from Wolfe (2024): "lower temperatures skew the judge's output towards lower scores." This is because lower temperature concentrates probability mass on the most likely tokens, which for scoring tasks tends to be more conservative/central values. This is worth monitoring during calibration.

### Recommendation

**Set temperature to 0.** Our evaluation task requires consistency above all else. We want the same article to receive the same score across multiple runs. The slight conservative bias at low temperatures is acceptable and can be calibrated for in the scoring rubric.

If using self-consistency (running multiple evaluations and aggregating), a slightly higher temperature (0.3-0.5) with 3-5 samples would provide diversity while still being well-calibrated. But for single-pass evaluation, temperature 0 is the standard.

**Note:** Claude's API default temperature is 1.0. We are currently using this default, which is unnecessarily high for evaluation.

## 7. Model Selection

### Capability vs. Cost Tradeoff

**Pricing (as of early 2026):**

| Model | Input (per 1M tokens) | Output (per 1M tokens) | Relative Cost |
|-------|----------------------|------------------------|---------------|
| Claude Haiku 3.5 | $0.80 | $4.00 | 1x |
| Claude Sonnet 4 | $3.00 | $15.00 | ~3.75x |
| Claude Opus 4 | $15.00 | $75.00 | ~18.75x |

**The evidence on model selection for evaluation:**

- GPT-4 class models achieve "80% agreement rate with human preference scores, which matches the agreement rate that human annotators have with themselves" (Wolfe, 2024)
- "For many production workloads, Haiku's performance is sufficient" and Haiku can "process 3.75x more requests than Sonnet for the same cost"
- For text classification specifically, "frontier models are rarely needed -- GPT-4o-mini, Claude Haiku, and Gemini Flash all achieve 90%+ accuracy on well-prompted classification tasks"
- Simple binary classification tasks can use smaller models (7B-13B), while "complex tasks like nuanced reasoning assessment generally require larger, more capable models" (Patronus AI)

### Matching Model to Question Complexity

Our 8 scoring questions span a range of complexity:

**Binary questions (true/false) -- candidates for Haiku:**
- Q2: Novel framing (true/false)
- Q4: Author conviction (true/false)
- Q5: Practitioner voice (true/false)
- Q7: Named framework (true/false)

**Categorical questions -- need Sonnet:**
- Q1: Standalone passages (none/a_few/several/many) -- requires nuanced content analysis
- Q3: Content type (5 options) -- requires understanding document genre
- Q6: Content completeness (3 options) -- meta-assessment of content quality
- Q8: Applicable ideas (broadly/narrowly/not_really) -- requires reasoning about transferability

### The Case for Staying with Sonnet

While splitting binary questions to Haiku could save ~60% on those calls, the complexity cost is significant:
- Two API calls per article instead of one
- More code to maintain
- The binary questions benefit from context established while reading the full content
- The cost savings are modest in absolute terms (our scoring volume is low)

**Recommendation:** Stay with Sonnet for the unified scoring call. Consider Haiku only for the tagging task, which is simpler classification.

## 8. Batch Evaluation

### One Item Per Call vs. Multiple Items Per Call

**Research strongly favors one item per call for scoring tasks.**

Anthropic's own engineering guidance states: "It can help to create clear, structured rubrics to grade each dimension of a task, and then grade each dimension with an isolated LLM-as-judge rather than using one to grade all dimensions."

Evidently AI recommends: "If you have several aspects to evaluate, like completeness, accuracy, and relevance, it's best to split them into separate evaluators."

The key findings:

1. **One item per call** prevents cross-contamination between items. The model cannot be anchored by a previous item's quality when evaluating the next one.

2. **One dimension per call** keeps the model focused, reducing cognitive load. "LLM handles one quality at a time rather than dealing with complex reasoning."

3. **Multiple dimensions in one call** is acceptable when the dimensions are clearly separated and the model has sufficient capacity (Sonnet/Opus class).

### Separate Dimensions vs. Combined Scoring

The strongest recommendation from the literature is to evaluate each quality dimension separately. However, this must be weighed against practical costs:

- **Separate calls per dimension**: 4 API calls per article (one per scoring bucket). More accurate per dimension, but 4x the cost and latency.
- **Combined call**: 1 API call per article covering all 8 questions. Some cross-dimension interference, but massive cost savings.

**Our current approach (8 questions in one call) is a reasonable middle ground.** The questions are clearly separated and labeled, each has explicit answer options, and the model has enough capacity to handle them simultaneously. The research that recommends splitting is primarily about fundamentally different evaluation tasks (accuracy vs. helpfulness vs. safety), not about sub-questions within a coherent scoring rubric.

### What We Should NOT Do

Do not batch multiple *articles* into a single scoring call. Each article must be evaluated independently to avoid:
- Anchoring: The first article's quality influencing perception of subsequent articles
- Context window pressure: Multiple articles plus the rubric may exceed useful context
- Attribution confusion: The model mixing up which observations belong to which article

## 9. Self-Consistency

### Running the Same Evaluation N Times

Self-consistency is a technique where the same evaluation is run multiple times and results are aggregated. The premise is that if multiple independent reasoning paths converge on the same answer, that answer is more likely correct.

**Methods of aggregation:**

1. **Majority vote**: For categorical answers (our use case), take the most common response across N runs. Simple and effective.

2. **Mean/median**: For numeric scores, average the results. Reduces variance.

3. **Ranked voting** (Borda count, instant-runoff): More sophisticated than simple majority vote. Recent 2025 research at ACL proposes ranked voting based self-consistency that outperforms simple majority voting.

4. **Confidence-weighted**: Weight each response by the model's confidence (extractable from logprobs). Methods incorporating confidence scoring achieve cost reduction of up to 53% while maintaining accuracy.

**Panel of Diverse LLMs (PoLL):** Instead of running the same model multiple times, use 3 different smaller models as independent judges. Research showed this ensemble approach "outperformed single evaluators and cost one-seventh of using GPT-4 alone."

### When Self-Consistency is Worth the Cost

Self-consistency multiplies API costs by N (typically 3-5x). It's justified when:
- The cost of an incorrect evaluation is high (e.g., filtering out genuinely valuable content)
- The evaluation is binary/categorical (majority vote works cleanly)
- You observe high variance in single-run results

It's overkill when:
- Evaluations are low-stakes
- Single-run results are already stable
- Cost or latency is a primary constraint

### Known Limitations

Self-consistency can mask systematic errors: "when all samples converge on the same incorrect output ('self-consistent errors')." If the rubric is flawed or the model has a consistent blind spot, running the evaluation 5 times will produce the same wrong answer 5 times.

For long-context problems, "self-consistency can amplify position bias and correlated errors" rather than reducing them.

### Practical Recommendation

**Not recommended for our production scoring pipeline** due to cost multiplication and the already-acceptable consistency of single Sonnet calls. However, self-consistency is valuable during **calibration**: run each calibration article 3 times to identify questions where the model is inconsistent, which signals a rubric clarity problem.

## 10. Verbosity Control

### The Verbosity Bias Problem

LLM judges have a well-documented bias toward preferring longer, more verbose responses. "Verbosity bias is the inclination to rate longer, more detailed responses more favorably, even if a shorter response is more correct, concise, and helpful" (Evidently AI).

This bias manifests in two ways for our system:
1. **Input side**: Longer articles may receive higher scores simply because they contain more text for the model to latch onto, not because they have denser insight per unit length.
2. **Output side**: If we ask for explanations alongside scores, the model may produce longer explanations to justify higher scores, creating a self-reinforcing loop.

### Getting Answers vs. Getting Explanations

There is a tension between two goals:
- **Just the answers**: Minimizes token usage, faster, avoids the model talking itself into a score
- **Explanations + answers**: Enables debugging, supports CoT benefits, provides user-facing summaries

**Research supports requiring explanations** (see Section 1 on CoT), but with a critical constraint: the explanation should be generated *as part of the reasoning process*, not as a post-hoc justification.

### Our Current Approach

We ask for:
- 4 "brief reason" fields (quotability_reason, surprise_reason, argument_reason, insight_reason)
- 1 "overall_assessment" field (1-2 sentence summary)
- max_tokens=700

This is well-calibrated. The "brief reason" constraint keeps explanations short, and the reasons are genuinely useful for displaying to the user and debugging scoring decisions. The 700 max_tokens is sufficient for the JSON response without encouraging rambling.

### Recommendation

Keep the current verbosity level. The brief reasons serve the dual purpose of CoT (reasoning before scoring) and user-facing explainability. The key improvement is to ensure the prompt explicitly asks the model to reason *before* committing to the categorical answer for each dimension, rather than producing the answer first and the reason second.

## 11. Ordering Effects

### Position Bias Within the Prompt

Position bias -- the tendency to favor items based on their position in the prompt -- is one of the most studied biases in LLM evaluation.

**The landmark study** by Shi et al. (2024, published at IJCNLP-AACL 2025) evaluated 15 LLM judges across 150,000+ evaluation instances. Key findings:

- "Position bias is not a result of random variations" -- it is systematic
- Position bias is "weakly influenced by the length of prompt components but strongly affected by the quality gap between solutions"
- Different judges show different directional preferences (some favor first position, some favor last), and the direction varies by task
- GPT-4 demonstrated dramatic position effects: win-rates shifted "from 2.5% to 82.5% when outputs are switched from first to second position" in pairwise comparisons

### Does the Order of Questions Affect Answers?

**Yes.** Research on multi-criteria scoring found:

- Criteria like "Clarity" decreased in average score by approximately 3.5% when evaluated last versus first, indicating a **recency bias** within the prompt
- The evaluation context (holistic vs. isolated criteria) "dramatically impacts scores, with one model scoring an argument's 'Clarity' at 5.0/5 in isolation but 4.0/5 when scored holistically with other criteria"
- **Score rubric ordering** (ascending vs. descending score descriptions) also affects results

### Implications for Our 8-Question Prompt

Our questions are numbered 1-8 and always appear in the same order. This means:
- Q1 (Standalone Passages) may receive slightly more careful attention than Q8 (Applicable Ideas)
- The model may develop a "momentum" -- if early questions suggest high quality, later questions may be scored higher
- Questions placed earlier anchor the model's overall impression of the article

### Mitigation Strategies

1. **Randomize question order per call**: Not practical for us since the scoring maps depend on fixed question IDs, but worth noting
2. **Place the most important/nuanced questions first**: They get the most careful attention
3. **Make questions independent**: Ensure each question can be answered without reference to previous answers
4. **Use explicit anchoring language**: "Consider each question independently" in the prompt

## 12. Anchoring Effects

### How Reference Examples Bias Evaluations

Anchoring bias occurs when initial information (reference examples, prior scores, rubric framing) disproportionately influences subsequent judgments.

**Reference answer score bias** is a novel, previously unstudied type of scoring bias identified in recent research. When LLM judges are shown a reference answer marked as a "5/5," subsequent evaluations are pulled toward that anchor point. This is the LLM equivalent of the well-known human cognitive bias.

**Specific findings:**
- LLMs "exhibit familiarity bias and experience anchoring effects for multi-attribute judgments"
- Providing a reference answer marked with a specific score systematically shifts the distribution of subsequent scores
- The rubric's framing itself can anchor: describing "excellent" before "poor" produces different score distributions than "poor" before "excellent" (score rubric order bias)

### Anchoring in Our System

Our current prompt has several potential anchoring points:
1. **Word count**: Reporting word count before the content may anchor the model's expectations (longer = more substantive)
2. **Content warning**: When we note content is truncated, this may anchor toward lower scores regardless of content quality
3. **Question ordering**: Positive-framing questions early (Q1: "How many passages could stand alone?") may set a positive anchor before more neutral questions

### Mitigation Strategies

1. **Balance example quality**: If using few-shot examples, include equal numbers of high and low-scoring examples
2. **Randomize rubric order**: Present score options in random order (not always low-to-high)
3. **Isolate metadata from content**: Present the article content before metadata like word count
4. **Use counter-prompts**: "Evaluate for factual accuracy, then re-evaluate focusing only on whether the core question was answered, ignoring tone" (Kinde)
5. **Be explicit**: "Do not let the word count, author reputation, or publication name influence your evaluation of the actual content"

### Sources

- [Evaluating Scoring Bias in LLM-as-a-Judge](https://arxiv.org/abs/2506.22316) (2025)
- [Justice or Prejudice? Quantifying Biases in LLM-as-a-Judge](https://llm-judge-bias.github.io/) (2024)
- [LLM-as-a-Judge Done Right](https://kinde.com/learn/ai-for-software-engineering/best-practice/llm-as-a-judge-done-right-calibrating-guarding-debiasing-your-evaluators/) (Kinde)

## 13. Analysis of Our Current Approach

### What We Do Now

Our current scoring implementation (in `app/services/scoring_strategy.py`):

```python
model = "claude-sonnet-4-20250514"
response = anthropic_client.messages.create(
    model=model,
    max_tokens=700,
    messages=[{"role": "user", "content": prompt}],
)
```

**Prompt structure** (from `app/services/scorer.py`):
- Single user message (no system prompt)
- 8 categorical questions with fixed option lists
- Article metadata (title, author, word count) followed by content
- Requests JSON with categorical answers + brief reasons
- No few-shot examples
- No explicit CoT instruction
- Default temperature (1.0)
- Single pass (no self-consistency)

### What We Get Right

1. **Categorical over numeric**: Using categories like "none/a_few/several/many" instead of 0-100 numeric scores is well-supported. Binary and categorical evaluations are more reliable and consistent than fine-grained numeric scoring.

2. **Reasons alongside scores**: Asking for brief reasons serves as implicit CoT and provides debuggability.

3. **Clear question structure**: Each question has a number, label, description, and explicit option list. This is clear and unambiguous.

4. **Content-type-specific prompts**: Having different Q3/Q6 variants for articles vs. podcasts shows good domain specialization.

5. **Single item per call**: We never batch multiple articles into one scoring call.

6. **Reasonable max_tokens**: 700 tokens is sufficient for the JSON output without encouraging padding.

### What We Get Wrong

1. **Temperature = 1.0 (default)**: This is the most impactful issue. Every source recommends low temperature for evaluation. We should set `temperature=0`.

2. **No system prompt**: The rubric and scoring instructions should be in the system prompt, not crammed into the user message alongside the content.

3. **No explicit CoT instruction**: While we ask for reasons, we don't explicitly instruct the model to "think through your reasoning before committing to an answer." The reason fields come after the answer fields in the JSON structure for some dimensions, which encourages post-hoc rationalization rather than genuine reasoning.

4. **No few-shot examples**: Adding 2-3 calibration examples would improve scoring consistency, especially for the nuanced categorical distinctions.

5. **No score level definitions**: We ask for "none/a_few/several/many" standalone passages but never define what these mean quantitatively. "A few" to one person is 2-3; to another it is 4-5.

6. **Metadata ordering may anchor**: We present word count before content, potentially anchoring expectations. The content_warning for truncated articles may over-penalize.

## 14. Recommended Changes

### Priority 1: High Impact, Low Effort

#### 1a. Set temperature to 0

**Change:** Add `temperature=0` to the API call.

```python
response = anthropic_client.messages.create(
    model=model,
    max_tokens=700,
    temperature=0,
    messages=[{"role": "user", "content": prompt}],
)
```

**Expected impact:** Significantly improved scoring consistency. Same article should produce same score across runs. This is the single highest-ROI change.

**Evidence:** Universal recommendation across all sources. Current default of 1.0 is introducing unnecessary variance into every evaluation.

#### 1b. Move rubric to system prompt

**Change:** Split the prompt into a system message (rubric, instructions, format) and user message (article content).

```python
response = anthropic_client.messages.create(
    model=model,
    max_tokens=700,
    temperature=0,
    system=SCORING_SYSTEM_PROMPT,  # rubric + format instructions
    messages=[{"role": "user", "content": article_content}],
)
```

**Expected impact:** More consistent rubric adherence. Better prompt caching (system prompt is identical across calls). Cleaner separation of concerns.

### Priority 2: Medium Impact, Medium Effort

#### 2a. Add explicit score level definitions

**Change:** Define what each categorical option means quantitatively.

```
1. STANDALONE PASSAGES: How many passages could stand alone as a saved note?
   - none: No passages worth saving independently
   - a_few: 1-2 specific passages that are quotable or contain striking data
   - several: 3-5 strong passages with memorable phrasing or concrete examples
   - many: 6+ passages; the piece is densely packed with saveable content
```

**Expected impact:** Reduces ambiguity in the most subjective question. The model currently interprets "a_few" and "several" based on its training data priors, which may not match our intent.

#### 2b. Restructure JSON to put reasoning before answers

**Change:** Reorder the JSON fields so each dimension's reason comes before its categorical answer.

Current:
```json
{
  "standalone_passages": "several",
  "quotability_reason": "Contains 4 specific data points..."
}
```

Recommended:
```json
{
  "quotability_analysis": "The article contains 4 specific data points about...",
  "standalone_passages": "several"
}
```

**Expected impact:** Encourages genuine reasoning before committing to a categorical answer. Research consistently shows that rationale-before-score improves accuracy.

#### 2c. Add independence instruction

**Change:** Add to the system prompt: "Evaluate each question independently. Do not let your answer to one question influence your answer to another. A piece can be high on one dimension and low on another."

**Expected impact:** Reduces within-prompt anchoring and momentum effects.

### Priority 3: High Impact, High Effort

#### 3a. Add 2-3 few-shot calibration examples

**Change:** Include 2-3 scored example articles in the system prompt, covering the full score range.

```
=== CALIBRATION EXAMPLE 1 (High Score) ===
Title: "The Bitter Lesson" by Rich Sutton
Content: [first 500 words]
Scoring:
- standalone_passages: "many" (reason: "Multiple quotable claims about the nature of AI progress...")
- novel_framing: true (reason: "Reframes decades of AI research through a single lens...")
[...]
Total: 82/100

=== CALIBRATION EXAMPLE 2 (Low Score) ===
[...]
```

**Expected impact:** Calibrates the model's internal scoring scale to match our intent. Research shows this is "probably the single most helpful thing one could do to steer LLM generated outputs" (Confident AI). However, this requires careful selection and validation of examples, and increases input token costs.

**Caveat:** Must select examples carefully to avoid anchoring bias. Include examples across the full score distribution.

### Priority 4: Monitor and Consider

#### 4a. Self-consistency for calibration (not production)

During calibration/development, run each test article through the scorer 3 times and compare results. If a question produces inconsistent answers across runs, that indicates the rubric needs clarification.

#### 4b. Haiku for tagging

Consider switching the topic tagging task (in `app/services/tagger.py`) from Sonnet to Haiku. Tagging is a simpler classification task that doesn't require the nuanced reasoning of scoring. This would reduce tagging costs by ~75%.

#### 4c. Structured output mode

If JSON parsing failures become a problem, switch to Claude's structured output mode (constrained decoding). Currently the failure rate is low enough that this is not worth the added complexity.

### Changes NOT Recommended

1. **Splitting into separate API calls per dimension**: The cost multiplication (4x) and latency increase are not justified by the accuracy improvement for our use case. Our questions are well-structured enough to be answered in a single pass.

2. **Self-consistency in production**: Running 3-5 evaluations per article multiplies costs proportionally. Our single-pass Sonnet evaluations are consistent enough, especially after adding temperature=0.

3. **Switching to Opus**: The 5x cost increase over Sonnet is not justified for our evaluation task. Sonnet-class models achieve 80%+ agreement with human evaluators, which is sufficient.

4. **Randomizing question order**: Our scoring maps depend on fixed question IDs, and the implementation complexity is not worth the marginal reduction in ordering bias.

## Key Takeaways

1. **Temperature is the biggest quick win.** Changing from default (1.0) to 0 will immediately improve scoring consistency with zero code complexity.

2. **CoT works, and we partially have it.** Our "brief reason" fields provide implicit chain-of-thought, but reordering the JSON to put reasoning before answers will strengthen this effect.

3. **Few-shot examples are powerful but require careful curation.** They are the single most impactful prompt engineering technique, but poorly chosen examples can introduce anchoring bias.

4. **Our single-call, 8-question approach is defensible.** The literature recommends splitting dimensions into separate calls, but our questions are well-structured and the cost savings of a single call outweigh the marginal accuracy improvement of splitting.

5. **Position bias is real but manageable.** Our fixed question order introduces some bias, but the explicit option lists and clear question labels mitigate the worst effects.

6. **The rubric is more important than the examples.** Clear, unambiguous scoring criteria with defined score levels are the foundation. Everything else builds on top of this.
